OVERVIEW
================================================================================

Successfully implemented STORY-COMP-12, which adds electronic KYC (e-KYC) with
biometric verification capabilities to BANKO, complying with Circular 2025-06
regulatory requirements.

The implementation provides:
  - Multiple biometric verification types (facial recognition, fingerprint,
    document scan, video call, FIDO2/WebAuthn)
  - Confidence scoring (0.0 - 1.0 scale)
  - Liveness detection checks
  - Document verification with OCR support
  - Verification expiration management (configurable validity periods)
  - State machine for verification lifecycle
  - Comprehensive audit trail with timestamps

Key Features:
  - Verification status tracking (Pending -> InProgress -> Verified/Failed)
  - Automatic failure if confidence < 0.95 or liveness check fails
  - Expiration validation (biometrics valid for configurable period)
  - Domain-driven design with full separation of concerns
  - Complete test coverage with unit and integration tests

1. NEW DOMAIN ENTITIES: BiometricVerification (Domain Layer)
   Location: /crates/domain/src/compliance/entities.rs

   Enums:
   - BiometricType (FacialRecognition, Fingerprint, DocumentScan, VideoCall,
     Fido2WebAuthn)
   - BiometricStatus (Pending, InProgress, Verified, Failed, Expired)

   NewTypes:
   - BiometricVerificationId(Uuid) - type-safe ID wrapper

   Entity: BiometricVerification
   - Aggregate root managing verification lifecycle
   - Immutable by default, state changes through methods
   - Fields:
     * id: BiometricVerificationId
     * customer_id: Uuid
     * verification_type: BiometricType
     * status: BiometricStatus
     * confidence_score: f64 (0.0 - 1.0)
     * liveness_check: bool
     * document_type: Option<String> (e.g., "CIN", "Passport")
     * document_number: Option<String>
     * verified_at: Option<DateTime<Utc>>
     * created_at: DateTime<Utc>
     * expires_at: DateTime<Utc>

   Core Methods:
   - new(customer_id, verification_type, validity_days)
     -> Result<BiometricVerification, DomainError>
   - is_valid() -> bool (status == Verified && not expired)
   - mark_in_progress() -> Result<(), DomainError>
   - complete_verification(confidence, liveness, doc_type, doc_num)
     -> Result<(), DomainError>
     * Validates confidence in [0.0, 1.0]
     * Sets status=Verified if confidence>=0.95 AND liveness=true
     * Otherwise sets status=Failed
   - fail_verification(reason) -> Result<(), DomainError>
   - mark_expired() -> Result<(), DomainError>
   - Getter methods for all fields

   Validation Rules (applied in constructors/methods):
   - Validity must be positive (>0 days)
   - Confidence score must be in [0.0, 1.0]
   - Only valid state transitions allowed
   - Verification expires after configured period

   Tests (18 test cases in #[cfg(test)] block):

   - test_biometric_verification_invalid_confidence_score
   - test_biometric_type_display

3. NEW APPLICATION SERVICE: EkycService (Application Layer)
   Location: /crates/application/src/compliance/service.rs

   Constructor:
   - new(biometric_repo: Arc<dyn IBiometricRepository>) -> Self

   Main Methods:
   - initiate_verification(customer_id, verification_type, validity_days)
     * Creates pending verification
     * Stores in repository
     * Returns BiometricVerificationDto
     * Defaults to 90 days validity if not specified
     * Returns ComplianceError on invalid verification type

   - complete_verification(verification_id, confidence, liveness_passed,
                          document_type, document_number)
     * Retrieves existing verification from repository
     * Validates confidence score [0.0, 1.0]
     * Applies domain validation rules
     * Updates status: Verified (if confidence>=0.95 && liveness) else Failed
     * Sets verified_at timestamp on success
     * Returns updated BiometricVerificationDto
     * Returns ComplianceError on errors

   - check_verification_status(verification_id)
     * Returns current status string
     * Returns ComplianceError if not found

   - is_customer_ekyc_verified(customer_id)
     * Returns true if customer has valid (verified, not expired) biometric
     * Returns false if none exist or all expired

   - get_customer_verifications(customer_id)
     * Returns all verifications (any status) for customer

   - get_latest_verification_by_type(customer_id, verification_type)
     * Returns most recent verification of specific type

   - count_by_status(status)
     * For reporting (count how many pending, verified, failed, expired)

   Error Handling:
   - Uses ComplianceError enum for domain/application errors
   - Maps repository errors to RepositoryError variant
   - Maps invalid input to InvalidInput variant
   - Maps domain errors to DomainError variant

================================================================================
ARCHITECTURE DECISIONS
================================================================================

1. HEXAGONAL ARCHITECTURE
   - Domain entities in domain layer (banko_domain crate)
   - Application service in application layer
   - Repository port for persistence abstraction
   - Follows strict dependency direction: Infrastructure -> Application -> Domain

2. STATE MACHINE PATTERN
   - BiometricVerification implements explicit state transitions
   - Invalid transitions return DomainError
   - Only Pending can transition to InProgress
   - Only InProgress can transition to Verified/Failed
   - Expired status managed separately

3. VALUE OBJECTS & AGGREGATES
   - BiometricVerification is an aggregate root
   - BiometricVerificationId is a value object (newtype)
   - BiometricType and BiometricStatus are value objects (enums)
   - Domain entities apply validation in constructors

4. ERROR HANDLING
   - Domain validation via Result<T, DomainError>
   - Application validation via ComplianceError enum
   - Clear error propagation from layer to layer
   - No panics in production code

5. TESTING STRATEGY
   - Unit tests inline in domain entity (#[cfg(test)])
   - Integration tests in application service module
   - Mock repository for testing service without persistence
   - Tests verify both happy path and error cases

================================================================================
USAGE EXAMPLES
================================================================================

1. Initiating a Verification

   use banko_application::compliance::EkycService;

   let service = EkycService::new(Arc::new(repository));
   let verification = service.initiate_verification(
       customer_id,
       "facial_recognition",
       Some(90),
   ).await?;

   println!("Verification ID: {}", verification.id);
   println!("Status: {}", verification.status); // "pending"

2. Completing a Verification

   let completed = service.complete_verification(
       verification.id,
       0.98,           // confidence score
       true,           // liveness check passed
       Some("CIN".to_string()),
       Some("ABC123".to_string()),
   ).await?;

   println!("Status: {}", completed.status); // "verified"
   println!("Valid: {}", completed.is_valid); // true

3. Checking Customer e-KYC Status

   let is_verified = service.is_customer_ekyc_verified(customer_id).await?;
   if is_verified {
       println!("Customer has valid e-KYC verification");
   }

4. Getting All Verifications

   let verifications = service.get_customer_verifications(customer_id)
       .await?;

   for v in verifications {
       println!("{}: {} ({})", v.verification_type, v.status, v.confidence_score);
   }
