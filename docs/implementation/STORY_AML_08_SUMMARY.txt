OVERVIEW
================================================================================

Successfully implemented STORY-AML-08, which integrates the AML AssetFreeze
entity with the Account service to manage account freezes when suspicious
activity is detected.

When an account is frozen:
  - AssetFreeze record is created (for compliance/audit)
  - Account status is set to Suspended
  - Account available_balance is set to 0 (preventing transactions)
  - Total balance remains unchanged (for accounting)

================================================================================
WHAT WAS IMPLEMENTED
================================================================================

1. NEW PORT INTERFACE: IAccountFreezePort
   Location: /crates/application/src/aml/ports.rs

   Defines contract for account freeze/unfreeze operations:
   - find_account_by_id(account_id)
   - freeze_account(account_id)  [sets status=Suspended, available_balance=0]
   - unfreeze_account(account_id) [sets status=Active, restores available_balance]

   Purpose: Decouple AML context from Account context per DDD principles

2. ENHANCED AssetFreezeService
   Location: /crates/application/src/aml/service.rs

   New Methods:
   - with_account_port(freeze_repo, account_port) - constructor with port
   - freeze_account(account_id, reason, ordered_by)
     * Creates AssetFreeze record
     * Calls account_port.freeze_account() if available
     * Returns AssetFreezeResponse

   - lift_freeze(freeze_id, lifted_by)
     * Updates AssetFreeze status to Lifted
     * Calls account_port.unfreeze_account() if available
     * Returns updated AssetFreezeResponse

   - is_account_frozen(account_id) - check if account has active freeze

   Updated Methods:
   - new(freeze_repo) - now initializes account_port to None (backward compatible)

3. TEST COVERAGE
   Location: /crates/application/src/aml/service.rs (tests module)

   Added:

   - test_freeze_account_immediate() - basic freeze without port

KEY DESIGN DECISIONS
================================================================================

1. OPTIONAL PORT INTEGRATION
   - Service works with or without account port
   - Old code using AssetFreezeService::new() continues to work
   - New code can use with_account_port() for full integration
   - Errors from port don't fail freeze operation (graceful degradation)

2. SEPARATION OF CONCERNS
   - AML doesn't depend on Account service directly
   - Instead, defines port interface that Account context must satisfy
   - Infrastructure layer implements the adapter

3. IMMEDIATE FREEZE
   - No pending/approval state (per INV-09 requirement)
   - Freeze takes effect immediately
   - Only CTAF can lift the freeze (requires lifted_by parameter)

4. AVAILABLE_BALANCE vs TOTAL_BALANCE
   - Freeze sets available_balance to 0 (prevents new transactions)
   - Total balance unchanged (preserves accounting trail)
   - Unfreeze restores available_balance to match balance

5. HEXAGONAL ARCHITECTURE
   - Domain: AssetFreeze, Account (no external dependencies)
   - Application: AssetFreezeService, IAccountFreezePort trait
   - Infrastructure: Will implement IAccountFreezePort adapter (next step)

================================================================================
ARCHITECTURE FLOW
================================================================================

FREEZE OPERATION:
  Investigation → freeze_account() → Create AssetFreeze
                                  → Call account_port.freeze()
                                  → Account.freeze() method
                                  → Status = Suspended
                                  → available_balance = 0
                                  → Save both entities

LIFT OPERATION:
  CTAF Authorization → lift_freeze() → Update AssetFreeze (status = Lifted)
                                    → Call account_port.unfreeze()
                                    → Account.unfreeze() method
                                    → Status = Active
                                    → available_balance = balance
                                    → Save both entities

CHECK FROZEN STATUS:
  Any Operation → is_account_frozen() → Query active freeze
                                     → Return true/false
                                     → Use to gate transactions

QUESTIONS & ANSWERS
================================================================================

Q: What if account_port is None?
A: Freeze record is created but account isn't frozen. For testing/gradual rollout.

Q: Can multiple freezes exist on same account?
A: Yes, but only one can be ACTIVE. Multiple LIFTED freezes = historical records.

Q: What happens to pending holds when frozen?
A: Account can't apply new holds (status check fails). Existing holds remain.

Q: Can frozen account receive deposits?
A: No, deposit fails (account.deposit() checks status).

Q: When is unfreeze data recorded?
A: When lift_freeze() is called (lifted_at, lifted_by recorded).

Q: How to query all active freezes?
A: aml_service.list_freezes_for_account() returns all freezes for account.
   Filter by status "Active" to get currently frozen state.

Q: Is there an API endpoint?
A: Not yet. To be added in Phase 1 (infrastructure implementation).

Q: How do I know which account was frozen?
A: Check freeze.account_id in the response or database.
